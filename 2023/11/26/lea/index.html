1.背景

由于Flash必须写前擦除(即无法覆盖写),且擦除最小单元(Block)和写入最小单元(Page)粒度不一致,同时每个Block有一定的擦写(P/E)寿命限制, 致使Flash-based SSDs必须采用异地更新(Out-of-place Update)以及预留空间(Over-provisioning Space),磨损均衡等策略.

为了向使用SSD的Host端隐藏这些复杂的物理特性带来的限制,传统SSD会通过FTL(Flash Translation Layer)——传统SSD最重要的部件之一，来完成上述工作，即:

  • 地址转换(Address Mapping)
  • 磨损均衡(Wear-leveling)
  • 垃圾回收(Garbage Collection)

三项工作. 其中Address Mapping负责从逻辑地址到物理地址的映射.而映射表通常Cache在SSD自带的DRAM中,以便快速完成映射.

其中Address Mapping又根据映射粒度的不同分为三类:

  • Page Level
  • Block Level
  • Hybrid Level

后两类映射尽管拥有不错的空间效率，但会加剧写放大问题，致使读写性能相当差，故市场上主流现代SSDs都采用Page级映射.

Page级映射的最大问题就是空间开销大,传统SSD下Page级映射的每个条目大小为8 bytes(LPN:4 + PPN:4),在4 KB Page下，平均每1TB的SSD存储空间就需要1GB的空间来存储映射表，这样的空间开销对于SSD的DRAM来说太大了.因此只有部分映射表能被Cache到DRAM中.每当DRAM中不存在目标映射条目，就必须访问Flash内存寻找特定映射条目，这会带来很大的性能损失.

因此，主要的优化思路有以下两个:

  • 选取合适的替换策略，保证高频访问的映射条目都尽可能被Cache到DRAM中，减少Miss次数
  • 压缩映射表的空间大小，保证不损失过多查表性能的前提下把尽可能多的映射条目Cache到DRAM中

现有工作

DFTL

DFTL的思路非常简单，采用类似于TLB中LRU这样的缓存替换机制应用到FTL上.

首先所有的Mapping entry都会被持久化存储到Flash中，(专门存储Mapping entry的Page称为Translation Page).

我们把它们称为Global Mapping Table(GMT). 为了加快地址转换的速度，GMT中的部分条目会被Cache到SSD自带的DRAM中，我们把这部分Entry称为Cache Mapping Table(CMT).同时为了快速得到mapping entry在Flash中的位置, 由**Global Translation Directory(GTD)**来记录所有mapping entry的位置,通常都是全部Cache在DRAM中.

DFTL具体策略:

  • 每当需要进行一次地址转换,在CMT中查找，若找到则直接返回，否则根据GTD找到目标entry在Flash中的位置，从Flash中读出该entry，并Cache到CMT中，若CMT已满，则采用LRU策略腾空一个位置放入.
  • Lazy coping:直接更新CMT,等到被更改元素被换出时才对Flash中的GMT以及GTD进行更新.
  • Batch update:”Moreover, multiple valid data pages in the victim may have their virtual-to-
    physical address translations present in the same translation-page. By combining all these modiﬁcations into a
    single batch update, we reduce a number of redundant updates”

SFTL

SFTL在DFTL的基础上，在每次LRU换入时，将target entry所在的translation page中的所有entry全部换入(即增大换入换出粒度)，以最大利用空间局部性(spatial locality)。

同时注意到以Page为单位进行CMT的换入换出时，mapping entry在局部上呈现一定的线性性，因此可以通过记录线性段的开头entry以及一个bitmap来表示线性段的范围,可以节省一定的空间.

LeaFTL

(同样采用Page-level mapping，不过优化思路主要是在压缩映射表的空间上.采用学习型技巧，即贪婪分段线性回归.)

动机:

传统Page-level的Address Mapping的SOTA方法（DFTL/SFTL）一般都是human-driven heuristics的( i.e. 在替换策略下做功夫，力求尽可能满足程序的局部性原理,保证Cache miss尽可能减少以提高效率)，static的，无法runtime捕捉局部性特征。是否考虑通过引入Machine Learning的方法，我们可以更好地动态地捕捉到address mapping的特征.

设计思路:

1.主体设计

1.注意到,现代SSD通常会有一个Data Buffer，每当写入数据时，首先将写入请求Buffer起来，攒满一定大小(Block)后，一次性flush到一块连续(continuous)的物理空间中.

若我们将Buffer里的写入请求按照LBA进行排序，保证写入请求的LPA是单调递增的.同时为这个写入的Chunk分配的物理空间(PPA也是递增的),我们可以保证写入的LPA和与对应的PPA都是单调递增的.

2.将这些<LPA,PPA>在LPN-PPN图中标出，可以发现大部分点对呈现局部线性性

由此我们可以将一部分<LPA,PPA>条目用一个线性段segment来拟合表示,

segment记录形式为:<起始地址，segment长度，segment斜率, segment截距>(经优化后这样的一个segment仅仅需要8 bytes表示),

(例如<70, 10, 2.0, 3>表示LPA在[70, 80]间的LPA对应的PPA可以通过$PPA = f(LPA) = K *LPA +I = 2.0 *LPA + 3$计算而得)

而点对的线性拟合可以通过贪婪分段线性回归完成,算法时间复杂度为$O(N)$，空间复杂度为$O(1)$.

由此:

  • 在SSD写入前, 对Buffer的写入请求按照LPA进行重排,并进行分段线性回归,可以得到若干segment.
  • 单个segment大小为8 bytes, 可以覆盖多个<LPN,PPN>映射条目(每个4 bytes)（实验表明有98%的segment可以覆盖到至多128个<LPN,PPN>点对）
  • Segment可以节省空间,即便在最坏情况下,空间开销也不会比原来的Page-level更差
  • 我们通过segment,可以在$O(1)$内计算得到segment内的LPA对应的PPA.

我们可以用写入性能上的一些损失(重排/线性回归)换取Address Mapping映射表空间开销上的减少.

2.Misprediction(纠错机制)

$\gamma-approximate$的贪婪分段线性回归得到的segment存在两种misprediction.

  • 1.segment上的实际PPA与预测PPA有$[-\gamma,\gamma]$的误差

  • 2.segment上有”空洞”

**为什么允许misprediction?**一个松弛的Error bound可以保证segment的覆盖率更大

如何处理这类misprediction?

1.approximate segment计算出来的PPA实际上可能在实际PPA的$[-\gamma,\gamma]$范围内,我们可以在每个flash page的OOB区域存储page对应的LPA，每次计算得到PPA时访问对应的Page并检查其OOB中的反向映射信息，若正确则表面计算的PPA就是实际的PPA，否则我们需要在该不正确的PPA的OOB中存储在$[PPA-\gamma,PPA+\gamma]$范围内的PPA对应的正确LPA,在额外花费$\log(\gamma)$的查询代价就可以找到LPA对应的正确PPA.

2.我们并不清楚一个LPA是否真的属于某个approximate segment，故引入Conflict Resolution Buffer (CRB),即一个数组:

  • 每次插入approximate segment时,将approximate segment中的所有LPA按升序存储到数组中.
  • 不同approximate segment的LPA序列之间用null隔开.
  • 不同approximate segment的LPA序列按照每个approximate segment的start LPA按升序排列.
  • 整个CRB中没有重复的LPA，插入时进行去重，若插入的segment的start LPA与CRB中已有的某个start LPA 重复,将旧者+1.
  • approximate segment被删除时将所有元素从CRB中删去.
  • 每次查找LPA时，若发现为approximate segment，则进入CRB中二分查找到对应的LPA，并找到LPA对应的segment,并进行后续计算
3.segment的更新问题

直接更新segment，可能会导致:

  • 1.映射表的空间开销增大:

    例如:一个segment范围为[30, 70]，若[40, 60]需要Update，直接Update后变为:[30, 40], [40, 60], [60, 70]，若采用LSM-Tree结构,[30, 70]会被压入下一个level，仅有[30, 70], [40, 60]两项.

  • 2.性能损失(即需要额外访问内存),更新Block内Page状态表需要访问Flash来确认LPA对应的正确PPA,直接Update可能会带来性能开销

解决方案:采用LSM-Tree结构:segment更新时直接插入(不会造成额外的空间损失),不会造成直接的额外读,将Update所需的所有更新操作积攒起来在Compaction时一次性解决(Tail Latency反而更低?)

当且仅当Update的Segment将与其发生overleap的victim segment的所有LPA全部覆盖，才会将victim segment标记为invalid(需要额外访存更新相关meta data)，否则segment就是valid的(无需额外访存)，将其压入下一层level，等到后续主动compaction才会统一回收处理.若被压入下一层的segment依然发生overlap,则直接创建一个新的level将其放入其中，以防止递归下降.
